## Brief / 简介

This page records an internship learning track around Seamly2D, a Qt/C++ desktop application for clothing pattern design. The useful part is not a polished product claim, but the path from "can this project even build on my machine" to a working Qt6 environment, a small authentication add-on, and a repeatable release-packaging note.

这页记录的是南京图灵人工智能研究院实习期间留下的一段学习轨迹。内容不是把 Seamly2D 包装成一个完整产品，而是把当时真正花时间的部分摊开：Qt/C++ 怎么跑起来，qmake 项目怎么接上 Qt6，登录注册和用户管理怎么嵌进去，最后 Windows/macOS 的发布包怎么一步步整理。

## Development Shape / 开发形状

The feature work stayed deliberately small. Seamly2D itself is a large desktop codebase, so the safer move was to add a narrow authentication layer and keep the rest of the application path recognizable: first-run registration, login, control panel, administrator-only user management, and the original clothing-design entry.

功能改动没有故意扩大。Seamly2D 本身已经是一个很大的 Qt 桌面项目，当时比较现实的做法是只加一层窄的用户入口：首次注册、登录、控制面板、管理员用户管理，再回到原来的服装设计主流程。

The code side used `QDialog`, `QSettings`, salted SHA-256 password hashes, an `admin` first-user rule, input-method hints for username/password fields, and explicit role constraints for delete, promote, and demote operations.

代码侧主要接触了 `QDialog`、`QSettings`、加盐 SHA-256 密码哈希、首次用户必须为 `admin` 的规则、用户名/密码输入框的输入法限制，以及删除、升权、降权时的角色约束。

## Build And Release / 编译与发布

The harder part was not writing another dialog. It was understanding why a Qt desktop project succeeds on one machine and fails on another: Qt version mismatch, `core5compat`, MSVC toolchain selection, `windeployqt` not on PATH, missing `xerces-c_3_2.dll`, and the difference between a debug run inside Qt Creator and a releasable folder.

真正更耗时间的不是再写一个对话框，而是搞清楚为什么同一个 Qt 桌面项目在一台电脑上能跑、换一台就不行：Qt 版本不匹配、`core5compat`、MSVC 工具链、`windeployqt` 的 PATH、缺少 `xerces-c_3_2.dll`，以及 Qt Creator 里 Debug 能跑和 Release 文件夹能交付之间的差别。

## Review / 复盘

This internship trace is useful because it is close to how technical learning actually happens: first a vague task, then environment problems, then a small feature, then build failures, then packaging, and only later a clearer mental model of toolchains and release boundaries.

这段实习痕迹有价值，是因为它接近真实学习的样子：先是一个模糊任务，然后是环境问题，然后做一个小功能，再遇到编译失败和打包问题，最后才慢慢把工具链、发布边界和跨平台差异想清楚。
