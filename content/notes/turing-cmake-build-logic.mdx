## Early Idea / 前期想法

The build problem first looked like a list of commands: install CMake, open a developer prompt, run qmake, run build. That was not enough. When the error message changes after every small environment tweak, memorizing commands stops working.

编译问题一开始看起来像命令清单：装 CMake，打开开发者命令行，跑 qmake，点击编译。但这种记命令的方式很快就不够了。环境变量一改、工具链一换，报错也跟着变，只背步骤很难判断哪里错了。

## 改变

后来把问题拆成几层看：平台和架构先决定 x86/x64；MSVC、Windows SDK、`cl`、`link`、`lib` 负责把源码、目标文件、库和可执行文件连起来；CMake/Ninja 更像调度器，负责生成规则和按依赖图执行；Qt 则要能被 CMake 或 qmake 正确找到。

这样看之后，CMake 不是“另一个编译器”。它更像是把工具链、依赖、规则写清楚的人。真正编译和链接的还是底层工具。

![CMake to Make flow / CMake 到 Make 的流程图](/uploads/projects/nanjing-turing/cmake-make-flow.png)

## 阶段目标

- 分清 `cl`、`link`、`lib` 分别在做什么。
- 分清 `.cpp`、`.obj`、`.lib`、`.dll`、`.exe`、`.pdb` 的位置。
- 理解为什么 Qt 的架构、编译器版本和命令行环境要匹配。
- 理解 `CMAKE_PREFIX_PATH`、`Qt6_DIR` 和 `Qt6Config.cmake` 为什么会影响 Qt 包发现。
- 理解 Ninja 只是执行规则，不负责替代编译器。

## 复盘

这部分学习解决的是一个很现实的问题：以后看到“编译失败”，不要直接把它当成一个黑盒错误。先问是找不到工具、找不到头文件、找不到库、架构不一致、生成规则错了，还是运行时缺 DLL。能把问题归类，排查速度就会明显变快。
