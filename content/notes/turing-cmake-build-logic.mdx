## Early Idea / 一开始怎么想

At first, build problems looked like a command list: install CMake, open a developer prompt, run qmake, click build. That worked only until the next environment error. Once every small PATH or Kit change produced a different message, memorizing commands stopped helping.

一开始我把编译理解成命令清单：装 CMake，开开发者命令行，跑 qmake，点编译。后来发现这不够。环境变量一变、编译器一换、Qt 包路径一错，报错就换一套说法，只背步骤很难知道卡在哪里。

## How I Split It / 后来拆成几层

后来把它拆开看：

- 平台和架构先决定 x86 还是 x64。
- MSVC 里的 `cl` 负责编译，`link` 负责链接，`lib` 处理静态库。
- `PATH`、`INCLUDE`、`LIB` 决定命令、头文件和库能不能被找到。
- CMake 不是编译器，它负责 configure、generate，然后把规则交给 Ninja 或 MSBuild。
- Qt 还要能被找到，例如 `CMAKE_PREFIX_PATH`、`Qt6_DIR`、`Qt6Config.cmake` 是否指向正确安装目录。

这样看以后，“编译失败”就不再是一个黑盒。它可能是找不到工具、找不到头文件、找不到库、架构不一致，也可能是生成规则和 Qt 包版本不匹配。

## Notes From The PDF / PDF 里记了什么

The exported PDF is basically a learning map. It mixes diagrams, rough handwritten notes, and plain explanations of how source files become object files, libraries, executables, and finally something that still needs DLLs at runtime.

那份 PDF 不是教程成品，更像当时给自己画的一张地图。里面把 `.cpp`、`.obj`、`.lib`、`.dll`、`.exe`、`.pdb` 这些文件放在同一条链路里看，也记录了为什么 Qt 的架构、编译器版本和命令行环境要匹配。

## Looking Back / 回头看

这部分学习很基础，但很管用。以前看到编译报错容易直接复制错误信息去搜；现在至少会先问：这是配置阶段、生成阶段、编译阶段、链接阶段，还是运行时缺依赖？

能把错误归到某一层，排查就不再完全靠运气。
