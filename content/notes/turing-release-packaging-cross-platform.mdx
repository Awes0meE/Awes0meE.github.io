## Early Idea / 一开始怎么想

After the program ran inside Qt Creator, it was tempting to think the hard part was over. The Release step quickly corrected that. A program running on the development laptop is not the same thing as a folder someone else can double-click.

程序能在 Qt Creator 里跑起来以后，很容易以为“差不多了”。但 Release 打包马上把这个想法打掉：开发机能运行，不代表复制到另一台电脑上也能打开。

## Windows / Windows 侧

Windows 侧先要编 Release，再把 `seamly2d.exe` 放进干净目录，用 `windeployqt` 收集 Qt 依赖。只收 Qt 还不够，Seamly2D 还缺 `xerces-c_3_2.dll`，这个 DLL 不补上就会在别的机器上启动失败。

后面继续试了自签名证书和 Inno Setup。自签名只能解决本地测试和流程理解，不能当成正式发布证书。Inno Setup 的脚本则把一堆散文件变成安装包，至少让最终文件夹不再像临时复制出来的 build 目录。

## macOS / macOS 侧

macOS 侧换了一套麻烦：先把 Windows 上的代码移到 Mac，配置 Qt 和编译器，处理 Homebrew Qt 与官方 Qt 的路径差异，跑通 Debug，再切 Release。之后用 `macdeployqt` 收 frameworks，尝试自签名和 DMG。

签名这一步没有完全走到正式发布那种程度，但脚本把编译、打包、依赖收集这些部分尽量串了起来。除签名外，自动化流程基本能跑。

## Looking Back / 回头看

Release 打包最容易被低估。开发阶段的问题通常会直接报编译错误；打包阶段的问题经常是“我这里能跑，别人那里打不开”。这次最大的收获是把“能跑”拆细：依赖齐不齐，架构对不对，DLL 有没有漏，签名是什么状态，安装包里到底放了哪些文件。
