## Early Idea / 前期想法

After the program could run inside Qt Creator, it was tempting to treat the task as finished. The release step proved otherwise. A program that runs on the development machine is not automatically a program that can be copied to another machine.

程序能在 Qt Creator 里跑起来以后，很容易以为任务结束了。但 Release 阶段证明不是这样：开发机上能运行，不代表复制到另一台电脑上也能运行。

## 改变

Windows 侧的关键不是单独拿到 `seamly2d.exe`，而是用 `windeployqt` 收集 Qt 依赖，再手动补上 Seamly2D 需要的 `xerces-c_3_2.dll`。之后才轮到代码签名、安装包和交付体验。

macOS 侧同样不是把 `.app` 拷出来就完事。需要 `macdeployqt` 收集 frameworks，再考虑签名、扩展属性、Gatekeeper 和 DMG。自签名证书只能解决一部分本地验证问题，不能等同于商业发布证书。

## Stage Goal / 阶段目标

The practical target was a repeatable release checklist: build Release, collect dependencies, verify the application outside the IDE, sign when possible, and package the final folder or app bundle.

实际目标是把发布流程写成能复现的检查表：先编 Release，再收集依赖，离开 IDE 单独运行验证，能签名就签名，最后再做安装包或 DMG。

- Windows：Release build -> copy `seamly2d.exe` -> run `windeployqt` -> add `xerces-c_3_2.dll` -> test -> sign -> Inno Setup installer.
- macOS：Release build -> copy `.app` -> run `macdeployqt` -> remove stale signature/attributes -> optional self-sign -> verify -> create DMG.
- 不复制公开：`Windows_deliver_pack.zip`、`macOS_deliver_pack.zip`、`Seamly2D__Release/`、安装器和 DLL。
- 公开保留：流程 PDF 和两个 shell 脚本，方便说明当时打包逻辑。

## 复盘

Release 打包最容易低估。开发阶段的问题通常会给出编译错误；发布阶段的问题很多时候表现为“别人电脑打不开”。这次记录的意义，是把“我这里能跑”拆成更明确的条件：依赖齐不齐、架构对不对、签名策略是什么、交付物里到底包含哪些文件。
