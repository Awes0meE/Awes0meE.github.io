## Early Idea / 一开始怎么想

The first plan sounded simple: clone Seamly2D, open it in Qt Creator, and run it on Windows 11. The first real lesson was that "a Qt project" is not enough information. Qt version, MSVC version, Windows SDK, debugger, qmake, and deploy steps all get a vote.

一开始想得很简单：把 Seamly2D 拉下来，用 Qt Creator 打开，然后点运行。真正开始做才发现，Qt 项目不是一个按钮能解决的东西。Qt 大版本、MSVC、Windows SDK、CDB 调试器、`.pro` 文件、生成目录、部署工具，哪一个不对都会报错。

## What Changed / 后来怎么改

最早试过 Qt 5.15.2，但工程里已经有 Qt6 写法和 `core5compat` 相关依赖。为了兼容 Qt5 去改一堆源码不太现实，所以后来改成 Qt 6.5.3 + MSVC2019 64-bit Kit。

这个选择比“装了新版 Qt”更具体：Qt 包、编译器 ABI、x64 环境和项目配置要对上。旧的 `.pro.user` 也删掉了，因为里面存着上一台机器的路径和 Kit 信息，继续沿用只会把问题带回来。

## Practical Fixes / 实际修正

The useful fixes were small but connected: delete stale user config, choose the Qt6 kit, bind MSVC 2019 v142, run qmake before building, and avoid letting deployment steps interrupt Debug builds.

实际处理大概是这样：

- 删掉旧 `.pro.user`，让 Qt Creator 重新生成本机配置。
- 选择 `Qt 6.5.3 MSVC2019 64-bit`，不要让 Qt5 和 Qt6 混在一起。
- 手动确认 MSVC 2019 v142、Windows SDK 和 x64 命令行。
- Build 前先跑 qmake，让生成规则更新。
- 把 `windeployqt` 这类部署步骤放到 Release 里处理，Debug 阶段先保证能调起来。
- 清掉本地窗口几何配置和 warning 干扰，避免“能编译但一启动就奇怪”的问题。

## Looking Back / 回头看

这次跑通环境最大的收获不是会点 Qt Creator，而是开始知道 Qt Creator 只是入口。真正决定能不能跑的是背后的 Kit、编译器、SDK、qmake 规则和运行时依赖。

以后再遇到类似桌面项目，我会先看项目文件和工具链匹配，再动代码。很多时候不是源码坏了，而是电脑上的环境没有和项目说同一种话。
