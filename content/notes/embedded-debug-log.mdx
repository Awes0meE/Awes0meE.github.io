## Starting Point / 前期想法

Early board bring-up often starts with a very plain failure: no LED, no serial output, no visible reaction. The worst part is not the bug itself, but the lack of record. When the board suddenly works again, the reason is still unclear.

嵌入式调试最怕“玄学变好”。一次能跑不代表问题被理解了。电源、时钟、下载、串口、最小循环、外设测试都需要一步步写下来：哪一步通过，哪一步没通过，看到什么现象。

## Stage Goals / 阶段目标

- Check power and flashing path before suspecting application code.
- Use serial output or LED blinking as the minimum observable signal.
- Add and test one peripheral at a time.
- Record why a failure disappeared, instead of relying on memory.

## Question / 疑问：为什么要这么麻烦？

Many embedded problems look like software bugs but come from power, clock, pin multiplexing, wiring order, baud rate, soldering, or initialization order. Random code edits can make the system even harder to understand. A debug log forces the process back to evidence.

很多嵌入式问题看起来像软件，其实是供电、时钟、引脚复用、下载器、焊接、线序、波特率或者外设初始化顺序。调试记录的意义就是先证明硬件和基础链路是活的，再动业务逻辑。

## Record Template / 记录模板

- Power: input voltage, regulator output, and abnormal heating.
- Flashing: ST-Link / J-Link recognition and chip model.
- Clock: HSE / HSI / PLL configuration.
- Serial: TX/RX wiring, baud rate, and shared ground.
- Minimum loop: stable LED toggle or serial print.
- Peripheral: one new peripheral per test round.

## Reflection / 复盘

Debugging should not depend on sudden inspiration. A repeatable checklist is more useful than a lucky fix. Future powered-up projects should turn this template into real photos, logs, and measurements.
